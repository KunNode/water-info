# 第5章 系统详细实现

## 5.1 业务平台实现

### 5.1.1 平台整体实现结构

业务平台是系统的基础服务层，主要承担站点、传感器、观测数据、阈值规则、告警、用户权限以及审计日志等模块的业务实现。平台基于 Spring Boot 3.2.2 开发，采用分层结构组织代码，各业务模块基本遵循“实体类、数据传输对象、视图对象、Mapper、Service、Controller”的结构方式，实现了数据访问、业务处理与接口输出的职责分离。

在具体实现上，平台通过 Spring MVC 暴露 REST 接口，通过 MyBatis-Plus 完成数据库访问，通过 Spring Security 与 JWT 完成认证授权，通过 Flyway 管理数据库版本迁移，并通过 WebSocket 向前端推送实时告警信息。这种实现方式保证了平台服务在业务治理、接口规范和工程可维护性方面具备较好的稳定性。

### 5.1.2 站点与观测数据模块实现

站点管理模块主要用于维护各类监测对象的基础信息，包括站点编号、站点名称、站点类型、位置和运行状态等。系统支持水位站、雨量站、流量站、水库、闸门和泵站等多种站点类型，从而为多源监测数据接入提供统一的管理入口。平台通过站点管理接口完成站点的新增、查询、修改和删除，并通过统一数据结构向前端和 AI 服务提供站点信息。

观测数据模块主要用于管理监测值的接入和查询。系统支持批量上传观测数据，并提供历史查询与最新数据查询接口，以满足常规业务展示和 AI 风险分析需要。由于观测数据具有明显的时序特征，平台在数据库设计中结合索引优化实现按站点、时间范围和指标类型的查询支持，从而保证数据管理与后续分析的效率。

### 5.1.3 阈值规则与告警模块实现

阈值规则模块用于配置不同站点、不同指标的告警触发条件。平台支持阈值规则的新增、启用、禁用、修改和删除操作，为后续告警生成和风险识别提供规则基础。系统通过阈值规则模块将业务判断标准从代码中抽离出来，提高了规则维护的灵活性。

告警模块在实现层面引入了状态机设计思想，将告警状态分为 OPEN、ACK 和 CLOSED 三种状态。系统在生成新告警时，首先判断是否已存在相同站点、相同指标、相同级别且处于 OPEN 状态的告警；若存在，则更新其最近触发时间和提示信息；若不存在，则创建新的告警记录。这种设计能够避免重复告警记录的大量堆积，并更加符合实际业务处理逻辑。

在告警状态流转方面，系统仅允许 OPEN 状态告警进入 ACK 状态，允许 OPEN 或 ACK 状态告警进入 CLOSED 状态，从而保证告警处理过程具有明确边界。平台在告警确认和告警关闭时还会同步记录操作日志，并通过 WebSocket 向前端广播告警更新结果，使前端能够实时展示最新状态。

### 5.1.4 用户权限与审计模块实现

为了满足系统安全管理需求，平台实现了基于 JWT 的认证机制和基于角色的权限控制机制。用户登录成功后，系统返回包含角色信息的访问令牌；后续请求通过请求头携带 Bearer Token，由 JWT 认证过滤器在请求进入业务处理前完成身份校验。这样既简化了服务端会话维护，也能够较好地适应前后端分离架构。

在授权方面，平台围绕用户、角色、组织和部门等对象建立了 RBAC 权限模型，不同角色拥有不同接口访问权限和业务操作范围。与此同时，系统还实现了审计日志模块，对关键业务操作进行记录，为系统使用过程中的问题追踪、责任追溯和安全管理提供支撑。

## 5.2 AI 多智能体服务实现

### 5.2.1 AI 服务整体实现

AI 服务基于 FastAPI 构建，对外提供洪水应急查询、流式查询、预案查询、预案执行和会话查询等接口。服务启动时会完成数据库连接池预热和相关持久化表初始化，以确保系统在进入业务处理前具备基本运行环境。AI 服务在内部通过 LangGraph 组织多智能体工作流，并以共享状态作为节点间协同的核心载体。

在具体实现中，系统为防洪应急查询定义了统一的请求和响应模型。用户提交查询后，服务首先生成或复用会话标识，并构造包含用户请求、会话历史和执行计数等内容的初始状态；随后调用编译后的多智能体工作流执行任务；任务结束后，从最终状态中提取风险等级、风险评分、预案编号、预案名称以及动作数量等结果，并封装为统一响应返回。该处理方式使 AI 服务既能够支持自然语言交互，也能保持较稳定的接口结构。

### 5.2.2 多智能体工作流实现

AI 服务中的核心实现是基于 LangGraph 的多智能体工作流。系统将监督智能体、数据分析智能体、风险评估智能体、预案生成智能体、资源调度智能体、通知智能体、执行监控智能体、并行调度节点和最终响应节点注册为状态图中的处理节点，并以监督节点作为图的入口。监督节点根据状态中的 `next_agent` 字段决定下一步路由，若状态表明任务处理完成，则进入最终响应节点并结束整个流程。

共享状态采用结构化定义，包含会话信息、站点数据、概览数据、告警数据、阈值规则、风险评估对象、应急预案对象、资源方案、通知记录、执行进度及最终文本结果等字段。各智能体在执行过程中围绕该状态进行读取和更新，避免了复杂任务依赖非结构化中间文本导致的上下文混乱问题。通过这种方式，系统实现了任务阶段之间的自然衔接，也为后续结果持久化和前端事件展示提供了统一数据基础。

### 5.2.3 平台接口调用与数据访问实现

在数据获取方面，AI 服务同时采用了数据库直读和平台 API 调用两种方式。对于实时态势分析所需的站点、观测、告警和规则信息，系统通过数据库服务直接从 PostgreSQL 读取，以降低链路延迟，提高分析效率。对于需要遵循业务规则的操作，如告警确认、告警关闭等，则通过平台 API 客户端访问业务平台接口完成。

平台客户端内部封装了登录鉴权、请求头构造以及常用资源访问方法。客户端在首次调用时自动向平台登录接口发起认证请求，获取 JWT 令牌后缓存于本地；后续调用则在请求头中自动附带 Bearer Token，从而完成平台接口访问认证。该实现减少了 AI 服务重复处理认证逻辑的复杂度，也为多智能体调用工具提供了统一入口。

### 5.2.4 结果持久化与会话管理实现

为提升系统可用性和后续追踪能力，AI 服务在生成应急预案后会尝试将预案概要、动作列表、资源方案和通知记录持久化到数据库中。系统将预案持久化设计为“尽力而为”的非阻塞操作，即即便持久化失败，也不影响主查询结果返回。这样做的目的是在保障用户响应体验的前提下，提高结果留存能力。

在会话管理方面，系统通过会话服务将用户轮次对话存入 Redis，用于支持后续上下文延续和查询回溯。当用户再次发起同一会话下的请求时，AI 服务可读取已有历史信息并构建连续语境，从而增强多轮交互体验。这一设计使 AI 服务从单轮问答工具进一步演化为具备上下文感知能力的业务辅助系统。

## 5.3 前端管理端实现

### 5.3.1 前端总体实现

前端管理端基于 Vue3、Vite、Pinia 和 Element Plus 构建，采用组件化和路由化方式组织系统页面。根据业务需求，前端包含登录、首页总览、站点管理、传感器管理、观测数据管理、阈值规则管理、告警管理、地图展示、大屏展示和 AI 智能指挥等多个视图模块。前端通过 Axios 请求后端业务平台接口，并在请求拦截器中统一附加 JWT 令牌，以实现用户身份保持和接口安全访问。

从实现结构上看，前端不仅承担常规业务页面展示，还通过组合式函数封装了 WebSocket 与 SSE 等实时通信逻辑，使 AI 查询和告警推送具有更好的复用性和可维护性。

### 5.3.2 AI 智能指挥页面实现

AI 智能指挥页面是前端中最具代表性的功能页面之一。该页面由聊天面板、智能体时间线、风险状态面板、预案状态面板、当前告警面板和会话信息面板等多个组件组成，形成类似指挥控制台的综合交互界面。用户在页面中输入自然语言请求后，前端首先创建用户消息和空的助手占位消息，然后通过 SSE 发起流式查询请求。

在结果处理上，页面持续监听服务端返回的结构化事件和文本片段。对于 `session_init` 事件，前端初始化会话标识和开始时间；对于 `agent_update` 事件，前端更新各智能体在时间线中的状态；对于 `risk_update` 事件，前端更新当前风险等级展示；对于 `plan_update` 事件，前端更新预案名称、进度与状态；对于 `agent_message` 事件，前端通过打字机效果逐步展示各智能体的说明信息。与此同时，普通文本流则不断填充助手消息内容，从而形成兼具最终结果和过程可视化的交互体验。

### 5.3.3 告警实时展示实现

在告警页面中，前端通过封装的 WebSocket 组合式函数与平台服务建立长连接，并持续监听 `/ws/alarms` 推送消息。系统在连接建立成功后将连接状态置为已连接，在收到消息后自动解析 JSON 数据并写入前端消息列表；同时，为避免消息列表无限增长带来的内存风险，前端为消息数量设置了上限并在超出时移除最早记录。

当平台侧告警状态发生变化时，前端能够实时感知并刷新告警展示内容，从而减少人工刷新页面的需要。这种实现方式使前端能够及时反映监测异常和告警处置进展，提高了系统的实时监控能力。

## 5.4 关键功能实现说明

### 5.4.1 洪水应急查询与预案生成实现

洪水应急查询功能是系统核心功能之一，其实现过程大致包括请求接收、会话初始化、状态构建、多智能体执行、结果提取、持久化和响应返回几个阶段。首先，系统接收用户提交的自然语言查询；其次，生成会话标识并加载可能存在的历史记录；然后，构造包含用户请求和上下文信息的初始状态，并交由多智能体图执行。工作流完成后，系统从状态中提取最终响应文本、风险等级、预案编号和资源数量等信息，封装后返回给调用方。

若用户通过流式方式发起请求，系统则会在处理过程中持续返回结构化事件与文本片段，使前端能够同步展示处理进度和中间结果。这种实现方式兼顾了业务结果输出和过程透明展示，是本文系统区别于传统静态问答式系统的重要体现。

### 5.4.2 告警状态机与实时广播实现

在告警功能中，系统引入状态机机制来控制告警状态流转。新告警创建后默认进入 OPEN 状态，值班人员确认处理后进入 ACK 状态，问题处置完成后进入 CLOSED 状态。系统在每次状态变化前都会进行合法性检查，若状态流转不符合规则，则抛出业务异常，从而避免错误操作破坏业务流程。

在告警广播方面，平台在创建新告警或更新既有告警时，会将站点信息、指标类型、告警级别、提示信息和时间信息封装为结构化对象，通过 WebSocket 广播给所有在线客户端。这样，前端不仅能够展示最新告警，还可以同步更新已有告警状态，从而增强监测和处置过程的实时性。

### 5.4.3 AI 流式事件与过程可视化实现

为了增强用户对 AI 处理过程的理解，系统在流式查询中除返回最终文本外，还设计了结构化事件机制。服务端在会话开始时发送会话初始化事件，在智能体进入或完成任务时发送状态更新事件，在风险等级和预案进度发生变化时发送风险更新和计划更新事件，在需要解释中间过程时发送智能体消息事件。前端接收到这些事件后，分别更新会话区、时间线、风险展示和预案状态区，并通过打字机效果逐步显示说明文本。

这种“文本流 + 结构化事件流”的双通道实现方式，使系统具备了较强的过程可观测性。对于毕业论文中的系统展示而言，这种实现不仅增强了用户体验，也体现了多智能体工作流的工程表达能力。

## 5.5 本章小结

本章围绕系统详细实现展开，分别说明了业务平台、AI 多智能体服务和前端管理端的实现过程。业务平台完成了站点、观测数据、阈值规则、告警、权限控制和审计日志等核心功能，并通过状态机和 WebSocket 实现了告警闭环与实时广播。AI 服务基于 FastAPI 与 LangGraph 实现了多智能体协同工作流，并结合数据库直读、平台 API 调用、结果持久化和 Redis 会话管理完成智能决策链路。前端管理端基于 Vue3 实现了统一业务页面和 AI 智能指挥页面，并通过 SSE 与 WebSocket 支持实时交互。上述实现内容表明，本文系统已经完成从业务管理到智能应急辅助的关键功能落地。
