# 第2章 相关理论与关键技术

## 2.1 多智能体系统相关理论

### 2.1.1 多智能体系统的基本概念

多智能体系统（Multi-Agent System，MAS）是由多个具备感知、决策、交互和执行能力的智能体共同构成的分布式智能系统。与传统单体智能系统相比，多智能体系统更加适合处理目标复杂、环节众多、环境动态变化明显的任务场景[1]。在该类系统中，不同智能体通常具有明确的角色分工和局部功能，通过通信、协商与协同完成整体任务目标。多智能体系统的核心特征包括分布式自治、异构协作、可扩展性以及对复杂环境的适应能力[1-3]。

在复杂业务场景中，单一模型往往难以同时兼顾数据理解、逻辑推理、方案生成与过程控制等多重任务要求，而多智能体系统可以将大任务拆分为若干相对独立但相互关联的子任务，并分配给具备不同职责的智能体执行。例如，在防洪应急场景中，数据分析、风险评估、预案生成、资源调度和执行监控本质上属于不同类型的决策问题，使用多智能体进行角色划分更符合业务逻辑，也更便于系统维护与扩展。

### 2.1.2 多智能体协同机制

多智能体协同的关键在于如何实现任务分解、状态共享、消息传递和执行编排[2-4]。从运行方式来看，多智能体协同通常包括三类典型机制：一是串行协同，即前一智能体的输出作为后一智能体的输入，形成线性处理链；二是并行协同，即多个智能体围绕同一目标同时处理不同子问题；三是监督式协同，即由上层协调者或监督者根据当前状态动态决定下一步应该激活的智能体[3-6]。

近年来，大语言模型的发展使多智能体协同机制获得了更强的语言理解、规划和工具调用能力。基于大语言模型的智能体不仅能够生成文本结果，还能根据任务上下文决定是否调用外部工具、访问结构化数据或将中间结果传递给其他智能体[2-6]。其中，ReAct 思想强调“推理-行动”交替过程，使模型能够在推理过程中不断借助外部环境修正决策[5]；多智能体对话框架则进一步增强了复杂任务中的协同组织能力[6]。这些研究为本文系统中构建监督智能体、数据分析智能体、风险评估智能体和预案生成智能体等角色提供了理论基础。

### 2.1.3 多智能体系统在应急管理中的适用性

应急管理场景通常具有信息来源多样、事件演化迅速、任务链条长和协同参与方多等特点，因此天然适合引入多智能体系统[7-10]。已有研究表明，多智能体方法在灾害路径规划、应急资源调度、院前急救管理和复杂灾害场景响应中具有较好的应用效果[7-10]。与传统规则式系统相比，多智能体系统可以更加灵活地根据场景变化重组任务流程；与单一决策模型相比，多智能体系统更容易实现业务分工和闭环控制。

对于防洪应急业务而言，系统既要处理实时水雨情数据，又要根据风险等级生成对应预案，还要考虑人员、设备、物资和通知对象的联动安排。因此，将防洪应急问题建模为多智能体协同任务，不仅可以降低单一模型的任务负担，还能使系统结构与实际业务流程保持一致，为后续系统设计和实现提供清晰的技术路径。

## 2.2 大语言模型与 Agent 工作流技术

### 2.2.1 大语言模型驱动的智能体

大语言模型具备较强的自然语言理解、知识组织、上下文推理和文本生成能力，为复杂任务自动化提供了新的技术基础[2-4]。在智能体系统中，大语言模型通常承担任务理解、意图识别、推理规划和结果生成等核心职责。当系统将模型与外部工具、数据库查询接口、业务 API 以及状态管理机制结合后，模型便不再局限于文本生成，而能够参与更完整的任务执行流程。

不过，单一大语言模型在复杂场景中仍然存在上下文负载过大、角色边界不清晰和可控性不足等问题[2-4]。因此，当前研究越来越倾向于将大语言模型以智能体角色嵌入工作流，通过多智能体方式实现任务拆分与责任分离。这种方式一方面保留了模型在语言理解和推理方面的优势，另一方面通过角色约束和流程控制降低系统不确定性。

### 2.2.2 Agent 工作流与工具调用机制

Agent 工作流是指以任务目标为导向，对智能体执行顺序、状态传递方式和工具调用过程进行显式组织的运行机制[5-7]。在典型工作流中，智能体需要根据用户输入和中间状态，决定当前是继续推理、调用工具、交给其他智能体处理，还是直接生成最终结果。为了实现这一目标，系统通常会为智能体提供统一的工具接口，使其能够访问数据库、外部服务、业务 API 或检索模块。

在本文对应的项目实现中，AI 服务采用了基于工作流的多智能体协同方式。监督智能体首先对用户请求进行理解，并决定应调用的数据分析、风险评估、预案生成、资源调度、通知生成或执行监控等智能体节点。各节点围绕共享状态读写中间结果，从而形成完整的任务链条。这种设计相比简单的问答式调用更适合防洪应急场景，因为它能够明确区分“分析”“判断”“生成”“调度”和“跟踪”等不同业务阶段。

### 2.2.3 LangGraph 状态图编排机制

LangGraph 是面向智能体工作流的图式编排框架，其核心思想是将复杂任务表示为状态图，并通过节点、边和共享状态组织多智能体协同流程[7]。在 LangGraph 中，节点通常对应具体的智能体或处理单元，边用于描述执行流向，而共享状态则负责在各节点之间传递结构化中间结果。相较于简单链式调用，状态图方式能够更好地支持条件分支、循环回路、人工干预和多节点并行等复杂流程。

结合本项目实现，AI 服务中的工作流由监督节点、数据分析节点、风险评估节点、预案生成节点、资源调度节点、通知节点、执行监控节点以及最终响应节点组成。系统在共享状态中维护用户请求、监测数据、风险评估结果、预案对象、通知记录和执行进度等信息，监督节点依据当前状态中的 `next_agent` 字段决定下一步路由。该机制使系统能够根据实际业务情况动态切换处理路径，而不是依赖固定的顺序流程，从而增强了多智能体协同的灵活性和可解释性。

## 2.3 系统开发关键技术

### 2.3.1 Spring Boot 与后端业务平台技术

Spring Boot 是当前 Java 生态中应用广泛的后端开发框架，具备自动配置、模块化集成和工程化能力强等特点[8]。在本系统中，后端业务平台基于 Spring Boot 3.2.2 构建，主要承担站点管理、观测数据管理、告警管理、阈值规则配置、用户认证授权和审计日志等核心业务功能。系统同时集成了 Spring Security、参数校验、WebSocket、WebFlux、Actuator 和 OpenAPI 等组件，使其具备接口安全控制、实时通信、运行监控和接口文档生成能力。

在数据访问层面，业务平台采用 MyBatis-Plus 进行对象关系映射和通用数据库操作，并结合 Flyway 实现数据库版本迁移管理。通过该组合，系统既能保持业务代码结构清晰，也便于后期演进和部署维护。对于毕业设计类系统而言，这种分层设计方式能够较好地平衡开发效率、代码可维护性和业务扩展能力。

### 2.3.2 FastAPI 与 AI 服务技术

FastAPI 是基于 Python 构建的高性能 Web 框架，适合开发接口清晰、并发能力要求较高的智能服务[9]。本系统中的 AI 服务基于 FastAPI 实现，对外提供防洪应急查询、流式响应、预案查询和执行管理等接口能力。AI 服务内部结合 LangGraph 组织多智能体协同逻辑，并使用 Pydantic 定义配置项和结构化状态对象，从而保证输入输出格式的一致性。

在数据访问方面，AI 服务使用 `asyncpg` 直接读取 PostgreSQL 中的实时监测数据，以降低查询链路延迟；在会话与缓存方面，系统利用 Redis 管理会话状态和相关中间信息[10-11]。与此同时，为了保持业务规则统一，涉及业务写回的操作并不直接修改数据库，而是通过平台服务接口完成。这种“读直连、写经平台”的协同模式兼顾了响应效率和业务一致性，是本文系统实现中的一个关键技术选择。

### 2.3.3 Vue3 前端与可视化技术

前端管理端承担系统交互、数据展示和结果反馈等功能，是连接业务平台和 AI 服务的重要界面层。Vue3 提供了响应式编程模型和组件化开发方式，适合构建交互频繁、状态复杂的单页应用[14]。本系统前端基于 Vue3 与 Vite 实现，结合 Pinia 进行状态管理、Vue Router 管理路由，并使用 Element Plus 构建管理端界面。

在可视化层面，系统使用 ECharts 展示监测趋势、统计数据和风险变化信息，使用 Leaflet 实现空间位置与地图场景展示。对于 AI 服务的流式输出，前端通过封装 SSE 组合式函数接收文本片段和结构化事件；对于告警消息，前端通过 WebSocket 维持与平台服务的实时连接并接收推送结果。该技术组合能够满足防洪应急场景下“结果可视、状态可追踪、过程可交互”的界面需求。

## 2.4 数据存储与实时通信技术

### 2.4.1 PostgreSQL 与 Redis

PostgreSQL 是功能完善的关系型数据库管理系统，支持事务处理、复杂查询和较强的数据一致性保障[10]。本系统采用 PostgreSQL 存储站点信息、观测数据、告警记录、阈值规则、用户与权限信息等核心业务数据。由于防洪应急系统同时涉及基础主数据管理和高频业务查询，关系型数据库在结构化存储、约束维护和事务一致性方面具有明显优势。

Redis 作为高性能内存数据库，常用于缓存、会话管理和高频读写场景[11]。在本系统中，Redis 既为业务平台提供缓存支撑，也为 AI 服务提供会话状态存储能力。通过关系型数据库与缓存数据库的协同使用，系统能够在保证数据可靠性的同时提升访问效率，从而更好地支撑实时防洪业务。

### 2.4.2 WebSocket 与 Server-Sent Events

WebSocket 是一种支持全双工通信的网络协议，能够在客户端与服务器之间建立持续连接，适合实时告警、状态同步和消息推送等场景[12]。在本系统中，平台服务通过 WebSocket 向前端推送告警事件，前端在建立连接后可以持续接收新增告警和状态变化信息，从而提高监控页面的实时性。

Server-Sent Events（SSE）是一种基于 HTTP 的单向流式推送机制，适用于服务器向客户端连续发送文本片段或事件流[13]。由于 AI 预案生成过程具有阶段性和流式输出特征，系统使用 SSE 向前端返回增量文本和结构化事件，使用户能够在等待完整结果前先看到风险评估更新、智能体处理进度和预案生成中间状态。相较于传统同步请求方式，SSE 机制能够显著改善用户体验，并增强 AI 处理过程的可观测性。

## 2.5 本章小结

本章围绕本文系统的理论基础与关键技术进行了分析。首先，从多智能体系统的概念、协同机制及其在应急管理中的适用性出发，说明了将防洪应急业务建模为多智能体协同流程的合理性。其次，结合大语言模型、Agent 工作流和 LangGraph 状态图编排机制，分析了智能体工作流在复杂任务处理中的优势。最后，介绍了本系统在实现层面采用的 Spring Boot、FastAPI、Vue3、PostgreSQL、Redis、WebSocket 和 SSE 等关键技术，并说明了这些技术在业务平台、AI 服务、前端展示和实时通信中的作用。上述内容为后续系统需求分析、总体设计和详细实现奠定了技术基础。
